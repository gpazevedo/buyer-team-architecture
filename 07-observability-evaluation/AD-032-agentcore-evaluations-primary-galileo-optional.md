# AD-032 — AgentCore Evaluations Primary, Galileo Optional, Three Evaluator Types

**Theme:** Observability & Evaluation  **Catalog:** AD-32 · **Source PRD:** PRD-004 · **Status:** Accepted · **Related:** AD-33, AD-34, AD-35, AD-49

## Context

Different agent outputs require different kinds of judgement. Tone and strategic soundness are qualitative; classification and ranking accuracy are deterministic; format and policy adherence are structural. Each kind has a different cost and latency profile. Using one evaluation mechanism for all outputs would either overpay (LLM-as-Judge applied to a structural check) or underserve (a code check grading tone). Galileo offers a significantly cheaper per-evaluation rate (~97% cheaper than Bedrock-based LLM-as-Judge) but would introduce an external vendor dependency and trace egress if made the production primary.

## Decision

Adopt AgentCore Evaluations as the production primary, with three evaluator types matched to the question being asked:

- **LLM-as-Judge** (medium cost, seconds) for qualitative metrics — tone, rationale quality, strategic alignment.
- **Ground Truth** (low cost, milliseconds) for deterministic accuracy against golden datasets (Kraljic classification, bid ranking).
- **Code-based / Lambda** (lowest cost, milliseconds) for structural checks — schema, required fields, policy adherence.

Galileo is an optional, feature-flagged addition (`galileo_observability_enabled`, `galileo_protect_enabled` in DynamoDB system-config), recommended for development and staging iteration, with graceful fallback to AgentCore-only if unreachable.

## Alternatives Considered

- **Galileo as production primary.** Rejected: introduces an external vendor dependency and trace egress in production; the ~97% per-eval cost saving does not outweigh the dependency risk at production baseline.
- **LLM-as-Judge for all evaluators.** Rejected: structural and deterministic checks do not require an LLM; using LLM-as-Judge for everything overpays by orders of magnitude and adds latency unnecessarily.
- **Code-based / Lambda for all evaluators.** Rejected: deterministic code cannot grade qualitative output such as tone or rationale quality.

## Trade-offs

| Gained | Given up |
| --- | --- |
| Evaluation cost is contained by construction: structural and ground-truth checks carry most of the load at millisecond latency and lowest cost | LLM-as-Judge evaluators are still medium-cost; their concentration in the qualitative tier directly motivates AD-33's sampling strategy |
| No external vendor dependency or trace egress in production | Teams lose the ~97% per-eval Galileo cost saving in production (retained in dev/staging) |
| `galileo_protect_enabled` hard-defaults to `false` when config is unreachable (AD-49), keeping security posture monotonic under config failure | Galileo evaluation richness (Luna-2 agentic metrics, hallucination detection) is absent from the production baseline |

`galileo_protect_enabled` hard-defaulting to `false` on config unavailability (AD-49) means a config outage leaves Galileo protection disabled rather than preserving a possibly-stale enabled state — a deliberate safety-monotonic design choice.

## Results

Evaluation cost is contained by design, with structural and ground-truth checks carrying most of the load cheaply. Teams get a significantly cheaper iteration loop in non-production environments without exposing production to a third-party dependency. The cost of the qualitative tier (LLM-as-Judge) is concentrated in one evaluator, which directly motivates AD-33's tiered sampling for Communication Tone. AD-34 (closed-loop automated actions) consumes the scores produced by this evaluator set, and AD-35 extends coverage with scheduled ATLAS-specific evaluators that run outside the per-trace online path.

**Implementation status as of PR #108 (2026-07-02): this evaluator set is mostly not built.** The AgentCore Evaluations subsystem this ADR names as production primary — LLM-as-Judge, Ground Truth, and Lambda evaluators wired to AgentCore Evaluations — was never built against real infrastructure. What exists today: the Ground Truth tier for Kraljic classification is implemented and CI-gated (`evals/dataset_coverage.py`, 200-row golden dataset, REQ-O219). The LLM-as-Judge tier exists only as a standalone offline stub (`evals/llm_judge.py`, Phase 0 "option c" — see [AD-113](AD-113-phase-0-eval-stub-scope.md)) that calls Bedrock directly; it is not the AgentCore Evaluations subsystem, is not deployed, and is not CI-gated. No Galileo integration has been observed built. `run_all.py`'s docstring (REQ-O224) is the current source of truth for what is CI-gated versus not.

**Update 2026-07-02 (deferred-scope build-out, Phase 2): the code-based / Lambda tier now has 2 real evaluators.** Bid Format Compliance (`evals/bid_format_compliance.py`) and Governance Policy Compliance (`evals/governance_policy_compliance.py`) are both CI-gated (100% pass on known-good, 100% fail on known-bad fixture rows) and inline-scored against live decisions: Bid Format Compliance and 5 of Governance Policy Compliance's 6 dimensions (budget ceiling, approved suppliers, ESG minimum, spend thresholds, max rounds) at `orchestrator/node_bid_evaluation.py` (`source="bid_evaluation_agent"` only — never the fallback stub); the 6th dimension (confidentiality) at `orchestrator/node_award_comms.py` against the persisted rejection-notification text (`source="award_comms_agent"` only). Both emit `procurement/evaluation` metrics via `emit_metric`, no alarm/SNS wiring (still deferred — AD-034's actual closed loop is explicitly out of scope for this build-out). The remaining gap in this tier: no AgentCore Evaluations registration — these are plain Python functions called inline, not evaluators registered with the AgentCore Evaluations subsystem this ADR names as primary.

**Update 2026-07-02 (deferred-scope build-out, Phase 3): the Ground Truth tier now has 2 evaluators.** Bid Ranking (`evals/bid_ranking_accuracy.py`) joins Kraljic Classification, CI-gated on a 100-negotiation golden dataset (`evals/fixtures/bid_ranking_golden.json`, `evals/bid_ranking_coverage.py`): mean Spearman rank-correlation >= 0.90 against a local composite-score baseline, plus zero top-1 mismatches on close-call negotiations. Unlike Kraljic, there is no independent DynamoDB ground-truth source for this tier — `{env}-bids` rows only carry the value the agent itself computed, not an expert label — so this evaluator is fixture-only, no live/dynamodb mode. No AgentCore Evaluations registration, same gap as Phase 2's Lambda tier.

**Update 2026-07-02 (deferred-scope build-out, Phase 4): the LLM-as-Judge tier now has 3 evaluators — state the split explicitly, don't blur it.** Award Rationale Defensibility remains the Phase 0 offline stub (`evals/llm_judge.py`, raw Bedrock `converse()`, untouched by this build-out). **Negotiation Quality and Communication Tone are new and Phoenix-backed** (`evals/phoenix_judge.py`, using `phoenix.evals.LLM(provider="bedrock", client="litellm")` — this is where Arize Phoenix replaces Galileo in the LLM-as-Judge implementation role for these two evaluators; live-verified against real Bedrock, not just fixture-shaped). Communication Tone is inline-scored at `orchestrator/node_award_comms.py` under AD-033's tiered-sampling gate (see AD-033 Results); Negotiation Quality is scored there too, against the award's `rationale` field (`source="bid_evaluation_agent"` only, no sampling gate — one score per negotiation, not per-message). `evals/judge_consistency.py`'s `dimensions`/`aggregate_field` were parameterized (previously hardcoded to Award Rationale's fields) so both new evaluators reuse REQ-O220's check instead of duplicating it. `evals/staging_suite.py` is a new, separate-from-CI entry point (`.github/workflows/staging-eval-suite.yml`, weekly) that runs both Phoenix judges against the 10-of-30 `panel_subset` transcripts for judge consistency + human calibration — none of this touches `run_all.py`'s offline PR-fast gate (still no AWS in that gate). Same gap as the other tiers: no AgentCore Evaluations registration — Phoenix is additive OTEL-native instrumentation, not a registration with the AgentCore Evaluations subsystem this ADR names as primary. Still no alarm/SNS wiring (AD-034 remains explicitly out of scope).

**Update 2026-07-12 (Phase C — AgentCore batch/A-B tooling for Kraljic Classifier, PR #199):** `evals/agentcore/batch_ab.py` adds Python tooling for batch evaluation and A/B comparison against AgentCore Evaluations, scoped to the Kraljic Classifier (the only agent with a registered AgentCore evaluator). `batch_ab.py` is a library, not a CI gate — it provides the primitives (`run_batch`, `compare_ab`) that a future CI workflow or scheduled evaluation can consume. The same structural gap persists: this is tooling that *targets* AgentCore Evaluations, not an evaluator registered *with* AgentCore Evaluations. No other agent has an AgentCore-registered evaluator to apply this tooling to.

**Update 2026-07-12 (Node 7 async self-invoke, PRs #202/#203):** the Phase 4 Communication Tone and Negotiation Quality evaluators (Phoenix-backed, inline-scored at `orchestrator/node_award_comms.py`) no longer run synchronously on the PO-issuance critical path. Node 7 now issues the PO first, then asynchronously self-invokes its own Lambda (`InvocationType=Event`, `{"_eval_only": True, ...}`) to run both judges out of process. A tight boto3 timeout with zero retries (PR #203) ensures a slow/throttled `lambda:Invoke` fails fast into the existing `except Exception` path rather than blocking. This removes ~10.8s of synchronous Bedrock judge latency from every award path. The scores are still emitted and still feed AD-034's closed-loop thresholds when those are wired — this change moves *when* they arrive, not *whether* they arrive. See AD-033 Results for the execution-model update.

**Update 2026-07-04: the code-based / Lambda tier gains 4 more evaluators, closing the negotiation-strategy-agent gap.** `bottleneck_negotiation`, `leverage_auction`, `spot_bidding`, and `strategic_partnership` previously had no dedicated evaluator of any kind — only the ATLAS adversarial-robustness replay (all 7 agents, see AD-35) and a wiring/registration test. This was tracked separately (`PRDs/agent-evals-overview.md`, not a REQ-O21x item — the Phase 2/3/4/5 build-outs above closed specific PRD-004 requirements, this closed a gap the requirement list never named). Four new evaluators (`evals/strategy_{bottleneck,leverage,spot,partnership}_quality.py` + matching `evals/fixtures/strategy_*_golden.json`), CI-gated in `evals/run_all.py --ci` on the same known-good/known-bad convention as Bid Format/Governance Compliance. Same self-referential-checks approach as Bid Ranking (Phase 3 above): there is no independent ground truth for any strategy agent's decisions, so each evaluator checks claims the agent's own response schema or system prompt already makes — e.g. bottleneck's recommendation must be the lowest risk-adjusted-TCO supplier, leverage-auction's prices must not increase round-over-round, spot-bidding's `response_rate` must equal its own stated ratio, strategic-partnership's `relationship_impact` (schema-required) must be populated alongside any recommendation. Same gap as every other tier: no AgentCore Evaluations registration, no live-AWS variant, no alarm/SNS wiring. Also explicitly does not touch AD-33's Communication Tone gap for these three negotiation-round agents (bottleneck/leverage-auction/strategic-partnership) — these are structural/logic checks, not tone judging; that gap remains open, tracked in AD-33's own Results.
