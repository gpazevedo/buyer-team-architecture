# AD-032 — AgentCore Evaluations Primary, Three Evaluator Types

**Theme:** Observability & Evaluation  **Catalog:** AD-32 · **Source PRD:** PRD-004 · **Status:** Accepted · **Related:** AD-33, AD-34, AD-35, AD-49

## Context

Different agent outputs require different kinds of judgement. Tone and strategic soundness are qualitative; classification and ranking accuracy are deterministic; format and policy adherence are structural. Each kind has a different cost and latency profile. Using one evaluation mechanism for all outputs would either overpay (LLM-as-Judge applied to a structural check) or underserve (a code check grading tone).

## Decision

Adopt AgentCore Evaluations as the production primary, with three evaluator types matched to the question being asked:

- **LLM-as-Judge** (medium cost, seconds) for qualitative metrics — tone, rationale quality, strategic alignment.
- **Ground Truth** (low cost, milliseconds) for deterministic accuracy against golden datasets (Kraljic classification, bid ranking).
- **Code-based / Lambda** (lowest cost, milliseconds) for structural checks — schema, required fields, policy adherence.

## Alternatives Considered

- **A third-party LLM-evaluation vendor as production primary.** Rejected: introduces an external vendor dependency and trace egress in production; a materially lower per-eval rate does not outweigh the dependency risk at production baseline. (An optional, feature-flagged dev/staging integration along these lines was evaluated and later removed from scope entirely as unbuilt — see Results.)
- **LLM-as-Judge for all evaluators.** Rejected: structural and deterministic checks do not require an LLM; using LLM-as-Judge for everything overpays by orders of magnitude and adds latency unnecessarily.
- **Code-based / Lambda for all evaluators.** Rejected: deterministic code cannot grade qualitative output such as tone or rationale quality.

## Trade-offs

| Gained | Given up |
| --- | --- |
| Evaluation cost is contained by construction: structural and ground-truth checks carry most of the load at millisecond latency and lowest cost | LLM-as-Judge evaluators are still medium-cost; their concentration in the qualitative tier directly motivates AD-33's sampling strategy |
| No external vendor dependency or trace egress at any environment tier | Evaluation richness that a specialized third-party eval platform might have added (agentic-metric detection, hallucination detection) is absent from the stack |

## Results

Evaluation cost is contained by design, with structural and ground-truth checks carrying most of the load cheaply. The cost of the qualitative tier (LLM-as-Judge) is concentrated in one evaluator, which directly motivates AD-33's tiered sampling for Communication Tone. AD-34 (closed-loop automated actions) consumes the scores produced by this evaluator set, and AD-35 extends coverage with scheduled ATLAS-specific evaluators that run outside the per-trace online path.

**Implementation status as of PR #108 (2026-07-02): this evaluator set is mostly not built.** The AgentCore Evaluations subsystem this ADR names as production primary — LLM-as-Judge, Ground Truth, and Lambda evaluators wired to AgentCore Evaluations — was never built against real infrastructure. What exists today: the Ground Truth tier for Kraljic classification is implemented and CI-gated (`evals/dataset_coverage.py`, 200-row golden dataset, REQ-O219). The LLM-as-Judge tier exists only as a standalone offline stub (`evals/llm_judge.py`, Phase 0 "option c" — see [AD-113](AD-113-phase-0-eval-stub-scope.md)) that calls Bedrock directly; it is not the AgentCore Evaluations subsystem, is not deployed, and is not CI-gated. No external evaluation-vendor integration was ever built against real infrastructure. `run_all.py`'s docstring (REQ-O224) is the current source of truth for what is CI-gated versus not.

**Update 2026-07-02 (deferred-scope build-out, Phase 2): the code-based / Lambda tier now has 2 real evaluators.** Bid Format Compliance (`evals/bid_format_compliance.py`) and Governance Policy Compliance (`evals/governance_policy_compliance.py`) are both CI-gated (100% pass on known-good, 100% fail on known-bad fixture rows) and inline-scored against live decisions: Bid Format Compliance and 5 of Governance Policy Compliance's 6 dimensions (budget ceiling, approved suppliers, ESG minimum, spend thresholds, max rounds) at `orchestrator/node_bid_evaluation.py` (`source="bid_evaluation_agent"` only — never the fallback stub); the 6th dimension (confidentiality) at `orchestrator/node_award_comms.py` against the persisted rejection-notification text (`source="award_comms_agent"` only). Both emit `procurement/evaluation` metrics via `emit_metric`, no alarm/SNS wiring (still deferred — AD-034's actual closed loop is explicitly out of scope for this build-out). The remaining gap in this tier: no AgentCore Evaluations registration — these are plain Python functions called inline, not evaluators registered with the AgentCore Evaluations subsystem this ADR names as primary.

**Update 2026-07-02 (deferred-scope build-out, Phase 3): the Ground Truth tier now has 2 evaluators.** Bid Ranking (`evals/bid_ranking_accuracy.py`) joins Kraljic Classification, CI-gated on a 100-negotiation golden dataset (`evals/fixtures/bid_ranking_golden.json`, `evals/bid_ranking_coverage.py`): mean Spearman rank-correlation >= 0.90 against a local composite-score baseline, plus zero top-1 mismatches on close-call negotiations. Unlike Kraljic, there is no independent DynamoDB ground-truth source for this tier — `{env}-bids` rows only carry the value the agent itself computed, not an expert label — so this evaluator is fixture-only, no live/dynamodb mode. No AgentCore Evaluations registration, same gap as Phase 2's Lambda tier.

**Update 2026-07-02 (deferred-scope build-out, Phase 4): the LLM-as-Judge tier now has 3 evaluators — state the split explicitly, don't blur it.** Award Rationale Defensibility remains the Phase 0 offline stub (`evals/llm_judge.py`, raw Bedrock `converse()`, untouched by this build-out). **Negotiation Quality and Communication Tone are new and Phoenix-backed** (`evals/phoenix_judge.py`, using `phoenix.evals.LLM(provider="bedrock", client="litellm")`, live-verified against real Bedrock, not just fixture-shaped). Communication Tone is inline-scored at `orchestrator/node_award_comms.py` under AD-033's tiered-sampling gate (see AD-033 Results); Negotiation Quality is scored there too, against the award's `rationale` field (`source="bid_evaluation_agent"` only, no sampling gate — one score per negotiation, not per-message). `evals/judge_consistency.py`'s `dimensions`/`aggregate_field` were parameterized (previously hardcoded to Award Rationale's fields) so both new evaluators reuse REQ-O220's check instead of duplicating it. `evals/staging_suite.py` is a new, separate-from-CI entry point (`.github/workflows/staging-eval-suite.yml`, weekly) that runs both Phoenix judges against the 10-of-30 `panel_subset` transcripts for judge consistency + human calibration — none of this touches `run_all.py`'s offline PR-fast gate (still no AWS in that gate). Same gap as the other tiers: no AgentCore Evaluations registration — Phoenix is additive OTEL-native instrumentation, not a registration with the AgentCore Evaluations subsystem this ADR names as primary. Still no alarm/SNS wiring (AD-034 remains explicitly out of scope).

**Update 2026-07-12 (Node 7 async self-invoke, PRs #202/#203):** the Communication Tone evaluator (inline-scored at `orchestrator/node_award_comms.py`) no longer runs synchronously on the PO-issuance critical path. Node 7 now issues the PO first, then asynchronously self-invokes its own Lambda (`InvocationType=Event`, `{"_eval_only": True, ...}`) to run it (and the confidentiality check) out of process — see `_run_eval_only`. A tight boto3 timeout with zero retries (PR #203) ensures a slow/throttled `lambda:Invoke` fails fast into the existing `except Exception` path rather than blocking. This removes ~10.8s of synchronous Bedrock judge latency from every award path. **Negotiation Quality is not part of this move** — per the 2026-07-06 correction below, it hadn't been scored on the live award path since PR #151, six days before this change. See AD-033 Results for the full execution-model update.

**Update 2026-07-04: the code-based / Lambda tier gains 4 more evaluators, closing the negotiation-strategy-agent gap.** `bottleneck_negotiation`, `leverage_auction`, `spot_bidding`, and `strategic_partnership` previously had no dedicated evaluator of any kind — only the ATLAS adversarial-robustness replay (all 7 agents, see AD-35) and a wiring/registration test. This was tracked separately (`PRDs/agent-evals-overview.md`, not a REQ-O21x item — the Phase 2/3/4/5 build-outs above closed specific PRD-004 requirements, this closed a gap the requirement list never named). Four new evaluators (`evals/strategy_{bottleneck,leverage,spot,partnership}_quality.py` + matching `evals/fixtures/strategy_*_golden.json`), CI-gated in `evals/run_all.py --ci` on the same known-good/known-bad convention as Bid Format/Governance Compliance. Same self-referential-checks approach as Bid Ranking (Phase 3 above): there is no independent ground truth for any strategy agent's decisions, so each evaluator checks claims the agent's own response schema or system prompt already makes — e.g. bottleneck's recommendation must be the lowest risk-adjusted-TCO supplier, leverage-auction's prices must not increase round-over-round, spot-bidding's `response_rate` must equal its own stated ratio, strategic-partnership's `relationship_impact` (schema-required) must be populated alongside any recommendation. Same gap as every other tier: no AgentCore Evaluations registration, no live-AWS variant, no alarm/SNS wiring. Also explicitly does not touch AD-33's Communication Tone gap for these three negotiation-round agents (bottleneck/leverage-auction/strategic-partnership) — these are structural/logic checks, not tone judging; that gap remains open, tracked in AD-33's own Results.

**Correction 2026-07-03 (PR #127): Negotiation Quality and Communication Tone are no longer Phoenix-backed.** The Phase 4 update above (and AD-33's Results) describe `evals/phoenix_judge.py` using `phoenix.evals.LLM(provider="bedrock", client="litellm")`. That module was removed one day after this update: `litellm`'s dependency chain (numpy, tokenizers, huggingface_hub) pushed `node_award_comms.py`'s shared node-Lambda deployment zip past the 50MB direct-upload limit. Both evaluators were rewritten as `evals/tone_negotiation_judge.py`, using a raw Bedrock `converse()` call instead — same scoring logic and thresholds, but no `phoenix.evals.LLM` wrapper and no auto-OpenInference span per judge call (the deployed ADOT layer still traces the Lambda invocation itself). `evals/staging_suite.py` and `orchestrator/node_award_comms.py` both import from `tone_negotiation_judge`, not `phoenix_judge` (which no longer exists in the repo).

**Correction 2026-07-06 (PR #151): Negotiation Quality is no longer scored live at all.** The Phase 4 update above says it's "scored there too, against the award's `rationale` field (`source="bid_evaluation_agent"` only)" — that source no longer exists. PR #151 removed the `bid_evaluation_llm` AgentCore agent entirely (replaced by deterministic inline scoring, `orchestrator/scoring.py`), and in the same change deleted `_score_negotiation_quality`/its call site from `node_award_comms.py` (repo-wide grep for `score_negotiation_quality` outside `evals/`/tests confirms zero call sites). Communication Tone is unaffected — it scores the notification text, not a `bid_evaluation_agent` rationale, and its call site survived. **Consequence for AD-34:** the `eval_negotiation_quality` CloudWatch alarm (`infra/modules/step-functions/eval_alarms.tf`, `metric_name = "negotiation_quality"`, `procurement/evaluation`) has watched a metric nothing emits live since 2026-07-06 — `treat_missing_data = "notBreaching"` means it silently sits in OK/insufficient-data forever rather than surfacing the gap. Negotiation Quality is still exercised, just never on a real award: `evals/judge_smoke.py` (every PR, 2 fixed fixture transcripts) and `evals/staging_suite.py` (weekly, `panel_subset` fixture transcripts) both still call `score_negotiation_quality`.

**Update 2026-07-11/12 (PR #197/#198/#199): the AgentCore Evaluations subsystem this ADR names as primary is now genuinely used — scoped to the Kraljic Classifier only, not the general evaluator set.** Every prior update in this Results section noted "no AgentCore Evaluations registration" as a remaining gap; that is no longer true for one agent. Three scoped phases, all under `evals/agentcore/`:

- **Phase A** (`evals/agentcore/builtin_kraljic_eval.py`, PR #197): on-demand `bedrock_agentcore.evaluation.client.EvaluationClient` run against the Kraljic golden-set sessions already produced by `orchestrator/eval_kraljic_accuracy.py --mode with_axes`, scoring `Builtin.Correctness` (vs. the seeded ground-truth quadrant) and `Builtin.ToolSelectionAccuracy`. Evaluator IDs are verified live via `list_evaluators()` before every run rather than hardcoded and trusted. Informational only, not CI-gated.
- **Phase B** (`infra/agentcore_evaluation/agentcore_evaluation.tf` + `.github/workflows/agentcore-online-eval-provision.yml`, PR #198): Terraform for `aws_bedrockagentcore_online_evaluation_config`, sampled online monitoring on the live Kraljic Classifier runtime, using `Builtin.Helpfulness` + `Builtin.ToolSelectionAccuracy` (not `Correctness` — the online-sampling API has no per-invocation reference-inputs field). **Built but not yet provisioned**: the provisioning workflow is `workflow_dispatch`-only by design (kept out of `dev-deploy.yml`'s automatic apply since the API is preview) and has zero runs as of this note — no live online-eval data exists yet.
- **Phase C** (`evals/agentcore/batch_ab.py`, PR #199): batch/A-B tooling (`BatchEvaluationRunner` + `CloudWatchDataSourceConfig`) reusing Phase A's two confirmed evaluators over the same golden set, for before/after comparison across a model-tier or prompt change. The original Phase C scope (porting `tone_negotiation_judge.py`'s two composites as custom evaluators for the transcript-panel A/B) was dropped — those run inline inside a Step Functions Lambda that emits no spans to a `/aws/bedrock-agentcore/runtimes/*` log group, the only span source the batch runner reads from.

Scope stays narrow: this is Kraljic Classifier only. The other 6 agents, the Lambda evaluators, and the two Phoenix-successor LLM-as-Judge evaluators remain unregistered with AgentCore Evaluations — the custom `evals/` package is still their real harness, per `PRDs/agent-evals-overview.md`.

**Update 2026-07-21: optional external evaluation-vendor integration removed from scope.** Earlier revisions of this ADR named an optional, feature-flagged dev/staging integration point with a third-party LLM-evaluation vendor (two `features`-group DynamoDB flags gating trace export and an agent-control plugin layer). Per the 2026-07-02 note above, no such integration was ever built against real infrastructure. The flags and every reference to that integration have been removed from this ADR (Decision, Alternatives, Trade-offs) and from PRD-001, PRD-004, PRD-005, PRD-006, PRD-008, and PRD-010. AgentCore Evaluations (Kraljic-scoped, per Phase A/B/C above) plus the Bedrock-direct `evals/tone_negotiation_judge.py` LLM-as-Judge implementation are the full, current evaluation stack — no third-party evaluation vendor is integrated or planned.
